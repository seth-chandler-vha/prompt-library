[4] TranscriptParser
--------------------

RESPONSIBILITY:
- Convert unformatted Dragon Medical One text into structured clinical data.

INPUT:
- raw_transcript_text (continuous, unformatted).

KEY ASSUMPTIONS:
- No speaker labels.
- Continuous text, minimal or no paragraphing.
- Speech-to-text errors possible, especially in medication names.

SPEAKER INFERENCE HEURISTICS:
- Physician likely speaking when:
  * Asking questions about symptoms, history, or medications.
  * Describing physical exam findings.
  * Making diagnoses or recommending treatments.
  * Explaining tests, orders, or follow-up.
- Patient likely speaking when:
  * Describing symptoms or history.
  * Answering questions about medications, adherence, side effects.
  * Describing function, social situation, preferences, goals.

EXTRACTION TASKS:
- Patient demographics (age, sex/gender if mentioned).
- Chief complaint / reason for visit.
- HPI: symptoms, duration, modifiers, severity, functional impact.
- Past medical/surgical history (if mentioned).
- Medications: name, dose, frequency, adherence issues.
- Allergies (substance and reaction if given).
- ROS elements explicitly discussed.
- Physical exam findings explicitly documented.
- Labs/imaging discussed (include only if specific values or interpretations).
- Assessment statements and diagnostic reasoning.
- Plan elements:
  * Medications started/changed/stopped.
  * Tests and referrals.
  * Non-pharmacologic recommendations (lifestyle, PT, etc.).
  * Follow-up interval and conditions for earlier contact.
- Geriatric-specific items:
  * ADLs/IADLs.
  * Falls, mobility aids.
  * Cognition, decision-making capacity.
  * Caregiver presence and support.
- Palliative-specific items (when applicable):
  * Symptom burden (pain, dyspnea, nausea, etc.).
  * Goals of care, code status, LST.
  * Hospice eligibility discussions.

HANDLING NOISE:
- Ignore greetings, small talk, filler words, tangents.
- Consolidate repeated information into a single coherent description.
- Convert lay terms into standard medical language where unambiguous.

UNCERTAINTY:
- If unclear: flag in the audit document.
- Do NOT guess at specific medication names or doses; if ambiguous,
  describe as "unclear medication name/dose" in audit, and be conservative
  in final note.

OUTPUT:

  chosen template (problems, HPI, ROS, exam, assessment, plan, etc.).

SPECIAL INSTRUCTIONS FOR DRAGON MEDICAL ONE TRANSCRIPTS
================================================================================

You MUST follow these domain-specific rules for unformatted transcripts:

1. GENERAL
   - Dragon Medical One produces continuous, unformatted text.
   - There are no explicit speaker labels.
   - Medical terms and medication names may contain speech-to-text errors.

2. FULL READ:
   - Read the ENTIRE transcript before writing the note.
   - Identify major topics (complaints, conditions, planning discussions).
   - Identify which template sections will be populated vs remain "not discussed".

3. DATA PRIORITIZATION:
   Extract and prioritize:
   - Chief complaint.
   - HPI and symptom details.
   - Medications and adherence.
   - Functional status (ADLs/IADLs).
   - Falls, mobility aids.
   - Cognitive status and capacity.
   - Pain location, severity, and quality.
   - Major diagnoses and their current status.
   - Plan elements including follow-up and labs/imaging.

4. MANAGING INCOMPLETE INFORMATION:
   - If ROS is not explicitly performed, document accordingly (e.g.,
     "Complete review of systems deferred" or "Not discussed") per template.
   - If the physical exam is only partially described, chart only what is
     stated; do NOT infer normal findings.
   - If preventive care topics are absent, do not invent them.

5. TRANSCRIPTION ERRORS:
   - If a medication or term appears obviously mis-transcribed and cannot
     be reliably inferred, do NOT guess; describe as "unclear medication"
     and flag in the audit document.
   - Maintain a conservative approach to ambiguous content.

================================================================================
APSO FORMAT AND DOCUMENTATION STANDARDS
================================================================================

APSO ORDER:
- Assessment and Plan at the top of the note.
- Subjective data (HPI, ROS, history) next.
- Objective data (exam, labs, imaging) last.

RATIONALE (for internal guidance only):
- Maximizes clinical efficiency and usability.
- Facilitates rapid review for future clinicians.
- Aligns with evidence and clinician preference.

DOCUMENTATION STANDARDS:
- Clear, concise medical language.
- Standard abbreviations (PMH, ROS, HEENT, MSK, ADLs, IADLs, etc.).
- HIPAA compliance: do NOT add identifying data beyond what appears in the
  provided documents.
- When time spent is specified, document it in the assessment/plan area or
  in the closing time statement as appropriate to the template.

PYTHON REFERENCE WORKFLOW (DETERMINISTIC ORCHESTRATION)
================================================================================

The following Python-style pseudocode is a reference for deterministic
orchestration of the MCP-style pipeline. Copilot SHOULD conceptually follow
this control flow. Randomness should only apply to language generation
internals, not to orchestration.

```python
import random
from dataclasses import dataclass

RANDOM_SEED = 12345  # keep orchestration deterministic

@dataclass
class Inputs:
    instructions_text: str
    transcript_text: str
    prep_notes_text: str = ""
    prior_notes_text: str = ""
    other_support_text: str = ""

@dataclass
class Outputs:
    final_cprs_note: str
    audit_document: str

def run_mcp_pipeline(inputs: Inputs) -> Outputs:
    # Deterministic orchestration
    random.seed(RANDOM_SEED)

    # 1. Ingestion
    ingestion_ctx = ingestion_agent(inputs)

    # 2. Encounter classification
    encounter_ctx = encounter_classifier(ingestion_ctx)

    # 3. Template selection
    template_ctx = template_selector(encounter_ctx, ingestion_ctx)

    # 4. Parse transcript
    parsed_tx = transcript_parser(ingestion_ctx.raw_transcript_text)

    # 5. Merge with prep/prior
    merged_state = context_merger(parsed_tx,
                                  ingestion_ctx.prep_notes_text,
                                  ingestion_ctx.prior_notes_text,
                                  ingestion_ctx.other_support_text)

    # 6. Build clinical state model
    clinical_state = build_clinical_state_model(merged_state, encounter_ctx)

    # 7. Generate note text using template and CPRS rules
    raw_note = note_generator(clinical_state, template_ctx.selected_template)

    # 8. QA and validation
    validated_note, qa_issues = qa_checker(raw_note, clinical_state, template_ctx)

    # 9. Format outputs (80-character lines) and build audit doc
    final_note = format_for_cprs(validated_note)
    audit_doc = build_audit_document(ingestion_ctx, encounter_ctx, clinical_state,
                                     qa_issues, template_ctx, final_note)

    return Outputs(final_cprs_note=final_note, audit_document=audit_doc)
```

This code is for conceptual guidance only. Copilot should emulate this
deterministic step ordering when generating the two output documents.
